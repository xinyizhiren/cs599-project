# ResearchFlow Product Spec

版本：v0.1  
日期：2026-06-05  
方向：方向一：Agentic AI 原生开发

## 1. 产品愿景

ResearchFlow 旨在构建一个证据可追溯的文献调研 Agent 系统。用户输入研究主题后，系统自动完成检索规划、论文检索、相关性筛选、证据抽取、跨论文综合、引用校验和调研报告生成。

产品目标不是替代研究者做最终学术判断，而是帮助研究者快速建立领域认知，并提供可复查的证据链。

## 2. 目标用户

| 用户 | 需求 | ResearchFlow 提供的价值 |
| --- | --- | --- |
| 研究生 | 快速进入新研究方向，准备开题和综述 | 生成结构化调研报告和核心论文列表 |
| 科研初学者 | 不熟悉检索关键词和领域脉络 | 自动拆解主题并扩展检索式 |
| 企业研发人员 | 快速判断技术路线和研究趋势 | 输出方法对比、研究空白和参考文献 |
| 课程项目评审 | 需要看到 Agentic AI 工程闭环 | 展示多 Agent、工具调用、状态流和评估 |

## 3. 问题定义

现有 AI 写作工具可以快速生成综述文字，但存在以下问题：

1. 检索过程不可见，难以复现。
2. 引用可能不真实或与结论不匹配。
3. 跨论文综合容易过度概括。
4. 结果质量缺少可量化评估。
5. 单轮 Prompt 难以稳定完成长流程调研任务。

ResearchFlow 将文献调研拆解为多个可检查节点，用 Agent 编排和工具调用形成完整闭环。

## 4. 产品范围

### 4.1 MVP 范围

MVP 聚焦命令行闭环：

- 输入一个英文或中文研究主题。
- 生成 3-5 组检索关键词。
- 调用至少一个开放论文数据源获取候选论文。
- 筛选 Top-K 论文。
- 抽取结构化证据。
- 生成 Markdown 调研报告。
- 保存任务 trace 和基础指标。

### 4.2 Final 范围

Final 阶段增加：

- 至少两个论文数据源。
- Citation Checker 引用校验。
- SQLite 记忆库。
- Benchmark 评估脚本。
- Docker 运行支持。
- 完整课程报告 PDF。

### 4.3 非目标

MVP 阶段不包含：

- 完整 Web UI。
- 大规模全文下载。
- 替代人工完成正式论文综述。
- 对付费数据库的集成。
- 对任意学科都达到专家级综述质量。

## 5. 用户故事

### US-001：生成文献调研报告

作为研究生，我希望输入一个研究主题，系统自动生成一份结构化调研报告，以便快速了解研究现状。

验收标准：

- 输入主题后能生成 Markdown 文件。
- 报告包含研究背景、核心论文、方法分类、研究空白和参考文献。
- 报告至少引用 5 篇候选论文。

### US-002：查看论文证据来源

作为用户，我希望报告中的关键结论能追溯到论文来源，以便判断内容是否可信。

验收标准：

- 每条关键结论关联至少一个 `evidence_id`。
- 每个 `evidence_id` 对应一条真实论文记录。
- 引用元数据包含标题、作者、年份、URL 或 DOI/arXiv ID。

### US-003：避免虚假引用

作为用户，我希望系统只引用检索到的真实论文，避免大模型编造参考文献。

验收标准：

- Report Writer 只能引用 Paper Store 中存在的论文。
- Citation Checker 能标记缺少 DOI/arXiv ID 或外部校验失败的引用。
- 校验失败的引用不得作为强证据输出。

### US-004：保存调研历史

作为用户，我希望保存调研过程和中间结果，以便后续复查或继续调研。

验收标准：

- 每次任务生成唯一 `task_id`。
- 系统保存检索式、候选论文、筛选论文、证据项、报告路径和基础指标。
- 失败任务也保留错误信息。

### US-005：评估系统质量

作为课程项目评审，我希望看到系统的功能测试和行为评估，以判断 Agent 是否可靠。

验收标准：

- 提供 benchmark 任务集。
- 每个 benchmark 输出检索成功率、引用有效率、报告完整率等指标。
- 测试结果可复现。

## 6. 功能需求

| ID | 功能需求 | 优先级 |
| --- | --- | --- |
| FR-001 | 系统应支持通过 CLI 输入研究主题和 Top-K 参数 | P0 |
| FR-002 | 系统应根据研究主题生成多组检索关键词 | P0 |
| FR-003 | 系统应调用论文检索工具返回候选论文 | P0 |
| FR-004 | 系统应对候选论文进行去重和排序 | P0 |
| FR-005 | 系统应抽取论文贡献、方法、实验、局限和结论 | P0 |
| FR-006 | 系统应生成 Markdown 调研报告 | P0 |
| FR-007 | 系统应记录每个 Agent 节点的输入、输出和耗时 | P1 |
| FR-008 | 系统应校验论文引用元数据 | P1 |
| FR-009 | 系统应保存任务历史到 SQLite | P1 |
| FR-010 | 系统应支持离线样例数据运行 | P1 |
| FR-011 | 系统应提供 benchmark 评估脚本 | P1 |
| FR-012 | 系统应支持 Docker 运行 | P2 |

## 7. 非功能需求

| ID | 非功能需求 | 说明 |
| --- | --- | --- |
| NFR-001 | 安全性 | 不得在代码中硬编码 API Key |
| NFR-002 | 可复现性 | 同一离线样例输入应产生稳定结构化输出 |
| NFR-003 | 可观测性 | 每个节点应有日志、状态和错误记录 |
| NFR-004 | 可扩展性 | 新论文数据源应能作为 Tool 插件接入 |
| NFR-005 | 可维护性 | Agent、Tool、Model、Storage 分层清晰 |
| NFR-006 | 稳定性 | 外部 API 失败时应支持重试、缓存或离线降级 |

## 8. 成功指标

| 指标 | MVP 目标 | Final 目标 |
| --- | --- | --- |
| 检索成功率 | >= 80% | >= 90% |
| 报告章节完整率 | >= 90% | >= 95% |
| 引用来自候选论文比例 | 100% | 100% |
| 引用元数据校验通过率 | MVP 可选 | >= 85% |
| Benchmark 任务数量 | >= 3 | >= 5 |
| 自动化测试 | 基础模型测试 | 覆盖核心工具和报告生成 |

## 9. 约束

- 课程最终仓库必须命名为 `cs599-project`。
- Public Repository 必须包含开源协议文件。
- 代码和文档不得硬编码 API Key。
- 最终报告 PDF 必须位于 `docs/CS599_大作业报告.pdf`。
- 最终报告必须包含封面、导航目录和课程要求的七个章节。

## 10. 风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| API 限流 | Demo 不稳定 | 缓存、离线样例、重试 |
| LLM 输出格式不稳定 | 节点失败 | Pydantic schema 校验和重试 |
| 文献相关性不足 | 报告质量低 | 多检索式召回和二次排序 |
| 引用幻觉 | 可信度下降 | Citation Checker 和引用白名单 |
| 时间紧张 | 无法完成 Final | MVP 优先命令行闭环，Web UI 放扩展 |
